Unique, standardized alphanumeric code representing the error category and specific issue. Format: `XXX.YYY.ZZZ` where each segment is uppercase and delimited by dots.
    - Example: `SYS.NET.AUTH.FAILURE`
- **numericCode**: *(integer)*
Numeric equivalent for internal referencing, between 10000–99999 for system errors, 20000–29999 for application errors, and 40000–49999 for user errors.
    - Example: `10001` for system shutdown error.
- **message**: *(string)*
User-friendly summary of the error.
- **developerMessage**: *(string, optional)*
Technical details for developers.
- **httpStatus**: *(integer)*
Relevant HTTP status code mapped to the error (see mapping below).
- **timestamp**: *(string, ISO 8601)*
Time error was generated or logged.
- **severity**: *(string)*
Severity level: `LOW`, `MEDIUM`, `HIGH`, or `CRITICAL`.
- **category**: *(string)*
High-level error category: `SYSTEM`, `NETWORK`, `VALIDATION`, `SECURITY`, `DATABASE`, `APPLICATION`.

***

### Numeric vs String Code Conventions

| Code Type | Format | Purpose | Range/Examples |
| :-- | :-- | :-- | :-- |
| String (errorCode) | Uppercase, dot-separated triple segment | Human-readable categorization | `SYS.NET.AUTH.FAILURE` |
| Numeric (numericCode) | Integer | Efficient indexing, legacy system integration | 10000–19999 System errors; 20000-29999 App errors; 40000-49999 User errors |

### Enforcement Steps and CI Checks

1. **Schema Validation**
On every pull request or automated log submission, validate error messages and responses against the JSON Schema to ensure required fields, types, and formats are correct.
2. **Linting for Code Format**
Apply linters in CI pipeline to enforce `errorCode` naming conventions and ranges for `numericCode`.
3. **HTTP Status Cross-Validation**
Implement checks to ensure the `httpStatus` value corresponds appropriately to the error `category` and severity.
4. **Automated Severity and Category Checks**
Guardrail scripts to flag inconsistent severity-level assignments (e.g., `CRITICAL` errors in `VALIDATION` category should be reviewed).
5. **Structured Logging Enforcement**
Ensure error logs stored in the repository or external databases are formatted according to the standard.
6. **Review Gate**
Pull requests touching error definitions must include approval from a designated reviewer specializing in error taxonomy.
7. **Documentation Upkeep**
Automatically generate error code indexes and documentation for internal and external use.
